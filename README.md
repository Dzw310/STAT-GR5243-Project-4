# LendingClub Loan Default Prediction

**STAT GR5243 — Applied Data Science, Project 4 (Final)**

An end-to-end machine learning pipeline that predicts whether a LendingClub loan will default, covering data cleaning, exploratory analysis, unsupervised learning, feature engineering, supervised modeling, and an interactive web application for real-time predictions.

**Live App**: [https://zd2372.shinyapps.io/loan-default-predictor/](https://zd2372.shinyapps.io/loan-default-predictor/)

## Problem Statement

LendingClub facilitated ~2.3 million loans between 2007 and 2018. Approximately 20% of completed loans ended in default ("Charged Off"), resulting in significant losses for investors. This project builds a predictive model to identify high-risk loans at origination, enabling better lending decisions and risk management.

## Dataset

- **Source**: [LendingClub Loan Data (2007-2018Q4)](https://www.kaggle.com/datasets/wordsforthewise/lending-club) via Kaggle
- **Raw size**: 2,260,701 loans with 151 features
- **After filtering**: 1,345,310 loans with terminal outcomes (Fully Paid or Charged Off)
- **Default rate**: 19.96%

The dataset also includes 27.6 million rejected applications (9 columns) used for comparative analysis.

## Pipeline Overview

### 1. Data Loading & Cleaning (`01`)

- Filters to terminal loan statuses (Fully Paid / Charged Off)
- Removes 81 columns: data leakage, IDs, free-text, high-null (>40%), joint-application, and redundant fields
- Fixes column formats (`int_rate`, `revol_util`, `term`, `emp_length`, `earliest_cr_line`)
- Handles missing values with a three-tier strategy:
  - **Sentinel fill** for "months since X" columns (NaN = event never happened)
  - **Direct fill** for known-meaning NaNs (`emp_length`) and low-null columns (<1%)
  - **Median fill** for remaining columns (3-13% missing)
- **Output**: 1,345,310 rows, 70 columns, zero nulls

### 2. Exploratory Data Analysis & Unsupervised Learning (`02`)

- Target distribution and temporal default rate trends
- Univariate/bivariate analysis of numerical and categorical features
- Correlation heatmap and feature-target correlation ranking
- Geographic analysis of default rates by state
- FICO vs. interest rate interaction heatmap
- Comparison with rejected loan applications
- **Unsupervised learning**:
  - **PCA**: 29 components capture 90% of variance; no clear linear separation in 2D
  - **K-Means** (k=5): Clusters differ meaningfully in default rate, income, and FICO
  - **t-SNE**: Local grouping visible but extensive class overlap confirms a hard classification problem

### 3. Feature Engineering (`03`)

- Merges `fico_range_low`/`fico_range_high` into `fico_avg`
- Creates domain-driven ratio features: `installment_to_income`, `loan_to_income`, `revol_bal_to_income`, `open_acc_ratio`
- Log-transforms skewed features (`annual_inc`, `revol_bal`, `tot_cur_bal`)
- Encodes categoricals:
  - `sub_grade`: ordinal (A1=1 ... G5=35)
  - `addr_state`: smoothed target encoding
  - `home_ownership`, `purpose`, `verification_status`: one-hot
- Drops near-zero-variance and highly correlated (|r| > 0.90) features
- Applies StandardScaler to numeric features
- **Output**: 73 features

### 4. Supervised Modeling (`04`)

**Temporal train/val/test split** (no future data leakage):
- Train: before July 2015 (613K loans, 17.9% default)
- Validation: July 2015 - June 2016 (213K loans, 20.1% default)
- Test: July 2016 onward (519K loans, 22.4% default)

**Leak-free preprocessing**: target encoding, NZV filter, correlation filter, and StandardScaler all fit on training data only.

**Models trained**:

| Model | Description |
|-------|-------------|
| Logistic Regression | ElasticNet baseline with `class_weight='balanced'` |
| Random Forest | 500 trees with balanced class weights |
| Neural Network | PyTorch MLP with BatchNorm, Dropout, and early stopping |
| XGBoost | Gradient boosting with `scale_pos_weight` tuning |

**Post-training pipeline**:
- Isotonic calibration on all models (class weights distort probabilities)
- F2-optimized decision thresholds (recall-weighted: missing a default costs more than a false alarm)
- SHAP analysis for model interpretability
- Application-time vs. investor model comparison
- Right-censoring sensitivity analysis for immature 60-month loans

**Top predictive features** (consensus across models): `sub_grade`, `term`, `fico_avg`, `loan_to_income`, `dti`, `revol_util`

### 5. Interactive Web Application (`app/`)

A Python Shiny app that serves the trained Logistic Regression and XGBoost models as an interactive loan default predictor, deployed on shinyapps.io. The Random Forest and Neural Network are excluded from the app: the calibrated Random Forest serializes to ~1.8 GB (deep trees with 500 estimators), exceeding the memory limits of free-tier hosting platforms, and the Neural Network requires PyTorch as a runtime dependency, which adds ~800 MB to the deployment bundle. Logistic Regression and XGBoost together total ~1.2 MB while covering both an interpretable baseline and the best-performing model.

**How it works**:

1. **Model artifacts** are exported from the last cell of NB04: calibrated model pickles, the fitted StandardScaler, target-encoded state mappings, NZV/correlation filter lists, F2-optimized thresholds, and dataset median values.

2. **User input**: The app exposes 12 tunable borrower parameters -- loan amount, term, annual income, employment length, FICO score, DTI, revolving balance, revolving utilization, credit age, loan purpose, home ownership, and state. The remaining ~60 features are held at dataset median values.

3. **Preprocessing pipeline** (replicates NB04 exactly):
   - User inputs are merged into a full 73-feature vector (medians for non-tunable features)
   - Derived features are recomputed: `loan_to_income`, `revol_bal_to_income`, `log_annual_inc`, etc.
   - `addr_state` is mapped to its smoothed target-encoded default rate
   - Categoricals are one-hot encoded (`home_ownership`, `purpose`, `verification_status`)
   - The vector is scaled using the training-set-fitted `StandardScaler`

4. **Prediction**: The scaled vector is passed to the selected model's `predict_proba()` method. The output probability is compared against the model's F2-optimized threshold to produce a binary HIGH RISK / LOW RISK decision. An explanation is shown describing why the threshold is low (cost-asymmetry: missing a default is far costlier than a false alarm).

5. **Display**: The results panel shows the default probability as a large gauge, the risk badge, a plain-language explanation, and comparison bars for both models.

## Key Findings

1. **Leak-free preprocessing is critical** -- StandardScaler, target encoding, and feature selection must fit on training data only
2. **Isotonic calibration benefits all models** -- class weights distort predicted probabilities across RF, LR, NN, and XGBoost
3. **Temporal splits respect deployment reality** -- shuffled CV leaks future data into training folds
4. **F2-optimized thresholds catch more defaults** -- missing a default is far costlier than rejecting a good loan
5. **Sub-grade is the single strongest predictor** -- it encodes LendingClub's own risk assessment (grades A-G, 5-35% default rate)

## How to Run

### Prerequisites

- Python >= 3.9
- ~4 GB free disk space for the raw CSV files
- ~8 GB RAM recommended (the accepted loans CSV is ~1.7 GB)

### Step 1: Clone the repository

```bash
git clone https://github.com/<your-username>/STAT-GR5243-Project-4.git
cd STAT-GR5243-Project-4
```

### Step 2: Create a virtual environment and install dependencies

```bash
python -m venv .venv
source .venv/bin/activate        # macOS / Linux
# .venv\Scripts\activate         # Windows

pip install -r requirements.txt
```

The `requirements.txt` includes: `pandas`, `numpy`, `scikit-learn`, `xgboost`, `torch`, `shap`, `matplotlib`, `seaborn`, and `jupyter`.

### Step 3: Download the data

Download the LendingClub dataset from [Kaggle](https://www.kaggle.com/datasets/wordsforthewise/lending-club) and place the two CSV files in the `data/` directory:

```
data/
├── accepted_2007_to_2018Q4.csv   (~1.7 GB)
└── rejected_2007_to_2018Q4.csv   (~1.8 GB)
```

### Step 4: Run the notebooks in order

```bash
jupyter notebook
```

Open and execute each notebook sequentially. Each notebook depends on the output of the previous one:

| Notebook | Description | Output |
|----------|-------------|--------|
| `01_data_loading_cleaning.ipynb` | Load raw data, filter target, clean columns, handle missing values | `data/cleaned.parquet` |
| `02_eda_unsupervised.ipynb` | EDA visualizations, PCA, K-Means, t-SNE | No output files (analysis only) |
| `03_feature_engineering.ipynb` | Ratio features, log transforms, encoding, scaling | `data/engineered.parquet`, `data/temporal_split_index.parquet` |
| `04_modeling_merged.ipynb` | Train 4 models, calibrate, optimize thresholds, SHAP analysis, export model artifacts | `app/models/*.pkl`, `app/models/*.json` |

### Step 5: Run the web app locally

```bash
pip install shiny
cd app
shiny run app.py
```

The app will be available at `http://localhost:8000`.

### Notes

- **Runtime**: Notebook 01 takes ~2-3 minutes (loading 2.3M rows). Notebook 04 takes ~10-15 minutes (hyperparameter tuning across 4 models).
- **GPU**: Not required. The PyTorch neural network trains on CPU. Set `device = 'cuda'` in Notebook 04 if a CUDA-compatible GPU is available.
- **Intermediate files**: All `.parquet` files are git-ignored. They are regenerated by running the notebooks.
